## Context

Stage 0 delivered the multi-tenant-safe kernel (fail-closed tenant isolation, auth,
RBAC, request context). Stage 1 builds the Phase 1 operational core on top of it, and
its **first slice (1.1)** is the physical model that nearly every later feature hangs
off: `Property ‚Üí Building ‚Üí Floor ‚Üí Zone`. Incidents, shifts, and patrols (1.5‚Äì1.7) all
reference zones/properties, so this hierarchy must exist and be correct first.

`Property` already exists from Stage 0 (with a single read-only `GET /api/properties`).
This slice adds the three lower levels, full CRUD, a cached property tree, cross-entity
validation, and the test coverage that extends the Stage 0 gate.

This is a **backend-only** slice. No admin UI is built now ‚Äî the real admin/onboarding
UI is Stage 2 by the project plan, and Stage 0 proved that backend correctness is best
established via tests + curl before any UI exists. The Enterprise "Command Center"
(portfolio dashboards, scores, heat maps, AI insights) is **Stage 4+**, gated on
customer demand; it is a downstream *consumer* of the data this slice and later slices
produce, not part of Stage 1.

### Decisions locked during brainstorming
1. **Scope:** API + data model + cache only. No UI this slice.
2. **Authorization:** admin-only writes; property-scoped reads via the existing
   `accessiblePropertyIds()` helper.
3. **Cache:** one assembled tree per property; bust that property's key on any write
   within it.
4. **Deletes:** never destroy data. Use soft-delete/archive (`archivedAt`) on all four
   tables ‚Äî a deliberate, documented deviation from the locked schema (which today only
   soft-deletes `Incident`).

---

## Goals / success criteria (the 1.1 Test Gate)

- Property-tree CRUD works single-tenant (build and read back the full hierarchy).
- Leak suite stays green with the new tables (every risky op, log-in-as-A ‚Üí zero-B).
- Cache invalidates on write (test).
- Zone-FK validation: a `zoneId`/`floorId` from a different property is rejected on
  write (A9 precursor).

Plus the recurring rules for every stage: tenant-scoped access only, fail-closed
filter intact, no raw SQL without the tenant-asserting wrapper, slow work off the hot
path, the leak suite still passes, allowlist unchanged.

## A. Data model (Prisma)

Add `Building`, `Floor`, `Zone`; add `archivedAt` + the new back-relations to
`Property`. Shapes follow `schema.md` tables 5‚Äì8, with `archivedAt` added per the C1
decision. One migration adds all of it.

```prisma
model Property {            // exists ‚Äî add archivedAt + back-relations
  // ...existing: id, tenantId, name, address, timezone, createdAt, tenant, userAccess
  archivedAt DateTime?      // NEW (C1 archive)
  buildings  Building[]     // NEW
  zones      Zone[]         // NEW
}

model Building {
  id         String   @id @default(cuid())
  tenantId   String                       // denormalized ‚Üí one-column isolation
  propertyId String
  name       String
  archivedAt DateTime?
  createdAt  DateTime @default(now())
  property Property @relation(fields: [propertyId], references: [id], onDelete: Cascade)
  floors   Floor[]
  @@index([tenantId, propertyId])
}

model Floor {
  id         String   @id @default(cuid())
  tenantId   String
  buildingId String
  name       String
  level      Int      @default(0)          // numeric so floors sort (B2=-2 ‚Ä¶ Ground=0)
  archivedAt DateTime?
  createdAt  DateTime @default(now())
  building Building @relation(fields: [buildingId], references: [id], onDelete: Cascade)
  zones    Zone[]
  @@index([tenantId, buildingId])
}

model Zone {
  id         String   @id @default(cuid())
  tenantId   String
  propertyId String
  floorId    String?                        // optional ‚Üí property-wide OR floor-level
  name       String
  archivedAt DateTime?
  createdAt  DateTime @default(now())
  property Property @relation(fields: [propertyId], references: [id], onDelete: Cascade)
  floor    Floor?   @relation(fields: [floorId], references: [id], onDelete: SetNull)
  @@index([tenantId, propertyId])
}
```

Notes:
- The DB `onDelete` rules remain as defense-in-depth, but the application **never
  hard-deletes** ‚Äî "remove" sets `archivedAt`.
- `tenantId` is always stamped from request context by the scoped Prisma client, never
  trusted from the client payload.
- These tables are tenant-scoped, **not** exempt ‚Äî the isolation allowlist stays
  exactly `{ Plan, SharedIntelligenceEntry }`.

## B. Code structure & endpoints

Extend the existing `apps/api/src/modules/properties/` module, following the Stage 0
house pattern exactly: `*.controller.ts` (thin), `*.routes.ts`, `*.service.ts` (logic
on `getScopedPrisma()`), Zod schemas wired via `validateBody`, `ok()` response
envelopes, `audit.record(...)` on writes. No new architectural layer ‚Äî the scoped
client is the "tenant-scoped repository" the checklist requires.

All endpoints sit under the existing tenant/context middleware and behind
`authenticate`. Writes additionally pass an admin role gate.

| Method | Path | Access | Purpose |
|---|---|---|---|
| `GET`   | `/api/properties`              | scoped read | list active properties (exists; add `archivedAt` filter) |
| `GET`   | `/api/properties/:id/tree`     | scoped read | cached `property‚Üíbuildings‚Üífloors‚Üízones` (active only) |
| `POST`  | `/api/properties`              | admin | create property |
| `PATCH` | `/api/properties/:id`          | admin | edit; archive via `archivedAt` |
| `POST`  | `/api/properties/:id/buildings`| admin | create building |
| `PATCH` | `/api/buildings/:id`           | admin | edit / archive |
| `POST`  | `/api/buildings/:id/floors`    | admin | create floor |
| `PATCH` | `/api/floors/:id`              | admin | edit / archive |
| `POST`  | `/api/properties/:id/zones`    | admin | create zone (optional `floorId`) |
| `PATCH` | `/api/zones/:id`              | admin | edit / archive / re-parent floor |

"Archive" is a `PATCH` setting `archivedAt`; archiving a parent archives its subtree
(logical cascade, rows retained).

## C. Invariants enforced in the service

The scoped client guarantees *tenant* isolation; cross-entity consistency within a
tenant is the service's responsibility:

1. **Parent exists & is active** ‚Äî creating under a parent loads the parent via the
   scoped client first: 404 if missing/cross-tenant, 409 if archived.
2. **Zone ‚Üî floor ‚Üî property consistency (A9 precursor)** ‚Äî if a zone carries a
   `floorId`, that floor's building must belong to the zone's `propertyId`; reject
   otherwise. This is the same-property rule `Incident.zoneId` / `ShiftAssignment.zoneId`
   will later depend on.
3. **Name uniqueness among active siblings** ‚Äî zone name unique within a property among
   non-archived rows (building name within property, floor name within building,
   likewise). Archiving frees the name for reuse.
4. **Archive cascade** ‚Äî archiving a node sets `archivedAt` on it and its descendants in
   one transaction; rows are retained.

Errors use the Stage 0 `AppError` codes (`BAD_REQUEST` 400 for invariant violations,
`NOT_FOUND` 404, `CONFLICT` 409 for archived-parent / name clash, `FORBIDDEN` 403 for
authz), surfaced through the existing error handler envelope.

## D. Caching (property tree)

- **Key:** `tenant:{tenantId}:proptree:{propertyId}` ‚Üí assembled active subtree as JSON.
- **Read path** (`GET /:id/tree`): check Redis; on hit return; on miss assemble from the
  scoped client (one query per level, stitched in memory ‚Äî no N+1), `SET` with a modest
  backstop TTL (‚âà1h), return.
- **Invalidation:** one helper `invalidatePropertyTree(propertyId)` `DEL`s that single
  key; called by every write touching the property or anything under it (creates, edits,
  archives, re-parenting). Re-parenting a zone stays within one property (invariant #2),
  so a single key is busted. Correctness comes from explicit busting; the TTL is only a
  safety net.
- Reuses `shared/redis/client.ts`. Only active nodes are cached, so archived data never
  appears in a cached tree.

---

## E. Authorization

- **Reads** (`GET` list + tree): any authenticated non-guard user; results filtered
  through the existing `accessiblePropertyIds()` ‚Äî admins see all tenant properties,
  managers/supervisors only their `UserPropertyAccess` grants, empty set ‚Üí empty/404.
  `:id/tree` 404s (not 403) when the property isn't in the caller's accessible set, to
  avoid revealing existence.
- **Writes:** gated to `HOTEL_ADMIN` / `SUPER_ADMIN` (includes impersonating platform
  staff, since impersonation mints a `SUPER_ADMIN` token). Uses the Stage 0 `rbac.ts`
  `requireRole(...)` helper as route middleware.
- Every write calls `audit.record(...)` (the Stage 0 audit seam): `property.create`,
  `building.archive`, `zone.update`, etc.

## F. Testing (extends the Stage 0 gate)

1. **Leak suite extended** ‚Äî Building/Floor/Zone added to the cross-tenant suite for
   every risky op (`findMany/findFirst/count/aggregate/groupBy/create/createMany/update/
   updateMany/delete/deleteMany/upsert`, nested writes, raw). Log in as A ‚Üí zero B rows.
   Allowlist-lock still asserts exactly `{ Plan, SharedIntelligenceEntry }`.
2. **Tree CRUD single-tenant** ‚Äî build property‚Üíbuilding‚Üífloor‚Üízone; read the tree back
   assembled correctly.
3. **Cache invalidates on write** ‚Äî read tree (caches), create/archive a zone, read
   again ‚Üí reflects the change.
4. **Zone-FK validation (A9)** ‚Äî a zone whose `floorId` belongs to a different property
   is rejected on write.
5. **Authz** ‚Äî a non-admin write is forbidden (403); a supervisor reads only granted
   properties; tree of an un-granted property 404s.
6. **Archive semantics** ‚Äî archived nodes drop from list/tree/cache; archiving a building
   archives its floors+zones (rows retained); an archived name can be reused.

Unit tests cover tree-assembly and the invariant helpers; integration tests drive the
real app via supertest (Stage 0 style), serially, against the migrated test DB. CI runs
the leak suite and blocks merge on failure (unchanged from Stage 0).

## G. Out of scope (this slice)

- Hard deletion of any hierarchy node (we only archive).
- Incident / shift / patrol FKs *to* zones (those tables arrive in 1.5/1.7 ‚Äî we build
  only the zone side now).
