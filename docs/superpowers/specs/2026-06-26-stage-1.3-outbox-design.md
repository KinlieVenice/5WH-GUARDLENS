# Stage 1.3 ‚Äî Transactional Outbox Design

**Date:** 2026-06-26
**Status:** Approved (brainstorm) ‚Äî pending implementation plan
**Stage:** 1.3 (third slice of Stage 1, the Phase 1 operational core)
**References:** `new-md/checklist.md` ¬ß1.3 ¬∑ `new-md/schema.md` table 11 (OutboxEvent) ¬∑ `new-md/architecture.md` ¬ß6 ¬∑ builds on Stage 0 + Stage 1.1 + Stage 1.2

---

## Context

The outbox is the **reliability backbone**: a state change writes an `OutboxEvent` row **in the
same transaction**, and a relay worker later drains those rows and fans them out (logbook lines,
notifications, follow-on jobs). This guarantees that "the thing happened" and "react to it" are
atomic ‚Äî a crash between them can never lose or duplicate the reaction.

The checklist marks the outbox's exactly-once behavior an **absolute test gate** (alongside the
Stage 0 leak suite): a failure there blocks everything after. So this slice is small in surface
but heavy on correctness proof.

No real producers or consumers exist yet (incident.created is 1.5; LogbookEntry is 1.6;
Notification is Stage 2). Therefore 1.3 builds the **mechanism** ‚Äî the table, the same-transaction
producer helper, a handler registry, and the relay loop ‚Äî and proves every gate property with a
**representative test consumer**. Real producers/consumers plug into the registry later.

There is no BullMQ in the project yet; the relay is its own poll loop (like the Stage 0
`token-cleanup` job), running inside `worker.ts`. BullMQ (for downstream dispatched jobs only)
arrives in a later stage.

### Decisions locked during brainstorming
1. **Dispatch scope:** build full outbox infra + a handler registry; ship **no** product
   consumers in 1.3; prove the gate with a representative TEST consumer that writes to a small
   `dedupeKey`-unique table created in test setup (product schema stays clean ‚Äî only `OutboxEvent`).
2. **Backoff:** exponential with a cap, env-configurable; dead-letter to `FAILED` after a max
   attempts count.

---

## Goals / success criteria (the 1.3 absolute gate)

- **Crash test:** kill the relay mid-fan-out ‚Üí on restart the event delivers **exactly once**,
  proven by the consumer's `dedupeKey` unique constraint rejecting the second write (not by timing).
- A rolled-back state change leaves **no** OutboxEvent row and **no** delivered effect (same-tx).
- A failing delivery backs off via `nextAttemptAt` and becomes `FAILED` after bounded `attempts`.
- Two concurrent relays never process the same row (`FOR UPDATE SKIP LOCKED`).
- Recurring rules: tenant-scoped access for app writes; the relay's cross-tenant reads go through
  the sanctioned base-client path; leak suite green with the new table; allowlist unchanged
  (`{Plan, SharedIntelligenceEntry}`).

## A. Data model (Prisma)

Add `OutboxEvent` exactly per table 11; add `outboxEvents OutboxEvent[]` to `Tenant`. One
migration. (`OutboxStatus { PENDING PROCESSED FAILED }` enum is also added.)

```prisma
enum OutboxStatus { PENDING PROCESSED FAILED }

model OutboxEvent {
  id            String       @id @default(cuid())
  tenantId      String
  type          String       // e.g. "incident.created"
  payload       Json
  status        OutboxStatus @default(PENDING)
  attempts      Int          @default(0)
  nextAttemptAt DateTime     @default(now())
  lockedUntil   DateTime?
  createdAt     DateTime     @default(now())
  processedAt   DateTime?
  tenant Tenant @relation(fields: [tenantId], references: [id], onDelete: Cascade)
  @@index([status, nextAttemptAt])
  @@index([tenantId, type])
}
```

Ordinary tenant-scoped model ‚Äî NOT exempt. App writes via the scoped client (tenantId stamped by
the extension); the relay reads/claims cross-tenant via the base client.

## B. Producer helper ‚Äî the same-transaction guarantee

`apps/api/src/modules/outbox/outbox.producer.ts`:

```ts
import type { Prisma } from "@prisma/client";
// Append an event in the SAME transaction as a state change. Callers do:
//   await db.$transaction(async (tx) => { ...write state...; await enqueueOutbox(tx, {type, payload}); });
// so the event and the state change commit atomically (crash ‚Üí neither).
export async function enqueueOutbox(
  tx: Prisma.TransactionClient,
  input: { tenantId: string; type: string; payload: unknown },
): Promise<void>;
```

`tenantId` is passed explicitly (the tx client is unscoped inside `$transaction`). No real producer
ships in 1.3; a representative producer exercises it in tests (write a row + an event in one tx;
prove rollback leaves neither).

## C. Handler registry

`apps/api/src/modules/outbox/outbox.registry.ts`:

```ts
export type OutboxEventView = { id: string; tenantId: string; type: string; payload: unknown };
export type OutboxHandler = (event: OutboxEventView) => Promise<void>;
export function register(type: string, handler: OutboxHandler): void;
export function getHandler(type: string): OutboxHandler | undefined;
```

1.3 registers no product handlers. An unknown type (no handler) is a delivery failure that backs
off/dead-letters like any other (so a missing consumer is visible, not silently dropped).

## D. Relay loop

`apps/api/src/modules/outbox/outbox.relay.ts` ‚Äî `runRelayOnce(): Promise<{processed:number; failed:number}>`,
driven on an interval by `worker.ts` (next to token-cleanup).

1. **Claim** (cross-tenant, base client, raw SQL ‚Äî Prisma's builder lacks SKIP LOCKED):
   inside a transaction, `SELECT id FROM OutboxEvent WHERE status='PENDING' AND nextAttemptAt<=NOW()
   ORDER BY nextAttemptAt LIMIT :batch FOR UPDATE SKIP LOCKED`, then set `lockedUntil = now + lockMs`
   on the claimed ids. SKIP LOCKED ‚áí concurrent relays never claim the same row.
2. For each claimed event, **re-establish tenant context** via `runWithContext({ tenantId })` and
   invoke `getHandler(type)`.
3. **Success** ‚Üí `status=PROCESSED, processedAt=now()`. **Throw / no handler** ‚Üí `attempts++`,
   `nextAttemptAt = now + min(base * 2^attempts, cap)`; when `attempts >= MAX_ATTEMPTS` ‚Üí
   `status=FAILED` (dead-letter). The outbox row is the single retry owner ‚Äî no stacked retries.

The relay imports `basePrisma` directly; the base-client-boundary leak test's sanctioned-importer
list is extended to include the relay (as it already includes `token-cleanup`).

## E. Exactly-once / dedupeKey contract

Marking `PROCESSED` and the handler's side-effect are separate writes, so a crash can land between
them. Idempotency is the **consumer's** responsibility, via a `dedupeKey` unique constraint on its
side-effect row, with `dedupeKey` derived deterministically from the event (e.g.
`${event.id}:${consumerName}`).

- Handler commits its side-effect ‚Üí relay crashes before `PROCESSED` ‚Üí row still `PENDING` ‚Üí next
  poll re-dispatches ‚Üí the handler's `dedupeKey` insert is **rejected by the unique constraint** ‚Üí
  no duplicate; relay then marks `PROCESSED`. Net: exactly one side-effect.
- Producer transaction rolls back ‚Üí no OutboxEvent row ‚Üí no delivery. All-or-nothing.

In 1.3 the representative test consumer writes to a small `dedupeKey`-unique table created in test
setup (`CREATE TABLE IF NOT EXISTS ‚Ä¶` via raw SQL), keeping the product schema to just
`OutboxEvent`. Real consumers (LogbookEntry 1.6, Notification Stage 2) carry their own `dedupeKey`'d
product tables.

## F. Config (env ‚Äî Stage 0 `config/env.ts` pattern)

- `OUTBOX_MAX_ATTEMPTS` (default 5)
- `OUTBOX_BACKOFF_BASE_MS` (default 1000)
- `OUTBOX_BACKOFF_CAP_MS` (default 300000)
- `OUTBOX_CLAIM_BATCH` (default 10)
- `OUTBOX_LOCK_MS` (default 30000) ‚Äî the `lockedUntil` lease window

Tests inject tiny values for speed/determinism.

## G. Testing (the absolute gate)

1. **Same-tx + rollback** ‚Äî a producer `$transaction` that throws after `enqueueOutbox` leaves
   **no** OutboxEvent row and **no** side-effect row.
2. **Happy path** ‚Äî `enqueueOutbox` + a registered handler ‚Üí one `runRelayOnce()` marks the row
   `PROCESSED` and the handler's side-effect exists exactly once.
3. **Crash exactly-once** ‚Äî handler commits its side-effect then "crashes" before `PROCESSED`
   (simulate by leaving the row PENDING); a second `runRelayOnce()` re-dispatches and the
   `dedupeKey` unique rejects the duplicate ‚Üí exactly one side-effect; row ends `PROCESSED`.
4. **Backoff ‚Üí FAILED** ‚Äî a handler that always throws: after each `runRelayOnce()` `attempts`
   increments and `nextAttemptAt` advances; after `MAX_ATTEMPTS` the row is `FAILED`.
5. **SKIP LOCKED** ‚Äî two `runRelayOnce()` calls racing one due row ‚Üí exactly one processes it
   (the other claims zero).
6. **Tenant context in relay** ‚Äî a handler runs a scoped query and sees only the event's tenant's
   rows (proves `runWithContext` re-establishment).
7. **Leak suite** ‚Äî add `outboxEvent` to `READ_MODELS` + seed one per tenant in factories
   (A‚ÜîB read isolation); allowlist-lock still exactly `{Plan, SharedIntelligenceEntry}`.

CI runs the suite serially against the migrated `hotelsec_test` DB; pristine output. The crash and
SKIP-LOCKED tests prove the gate by the `dedupeKey`/lock mechanics, not by timing.
