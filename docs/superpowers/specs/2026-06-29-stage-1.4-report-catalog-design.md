**Stage:** 1.4 (fourth slice of Stage 1, the Phase 1 operational core)
**References:** `new-md/checklist.md` ┬¦1.4 ┬Ę `new-md/schema.md` tables 12 (ReportType) & 13 (ReportTypeVersion) ┬Ę `new-md/module-list.md` module 2 ┬Ę builds on Stage 0 + Stage 1.1 + 1.2 + 1.3

## Context

Module 2 is the **report catalog**: a tenant-editable set of report types ("Theft",
"Hazard", "Medical") that drive incident creation (1.5). It is **not** one hardcoded
incident screen ŌĆö admins add and edit form fields without code, and each report pins the
**version** of the form it was filed against, so adding a field later never breaks an old
report.

The catalog is two tables: `ReportType` (the catalog entry ŌĆö name, routing lane, system
flag, active flag) and `ReportTypeVersion` (an immutable snapshot of the form's fields as
JSON). Editing a type's fields appends a **new version row** rather than mutating the old
one. The system ships six prebuilt types per tenant; admins can add their own and edit
either.

1.5 (Incident reporting) consumes this catalog ŌĆö every incident references a
`reportTypeId` + `reportTypeVersionId` and stores its answers in `fieldValues`. 1.4 builds
the catalog and its immutable versioning; it does **not** build incident filling or
fill-time field-value validation.

### Decisions locked during brainstorming
1. **UI scope:** API-only ŌĆö endpoints + service + field-schema validation + tests. No
   React this slice (consistent with 1.1ŌĆō1.3 backend-first). The form-builder UI is a
   later pass.
2. **Version trigger:** every field-schema save writes a **new** `ReportTypeVersion`
   (version = current max + 1); the old row is never mutated. `ReportType` metadata
   (`name`, `lane`, `isActive`) is patched in place and is **not** versioned.
3. **System seeding:** a code-defined, idempotent `seedSystemReportTypes(tenantId)`
   upserts the six prebuilt types + their v1. Called from the dev seed script and
   available for future tenant provisioning (Stage 2.1). No SQL-migration seed (the rows
   are per-tenant, keyed by `tenantId`).
4. **Field validation:** strict Zod schema on write ŌĆö type Ōłł {text, dropdown, photo,
   yesno}; unique non-empty `key` within a form; `label` required; `required:boolean`;
   `dropdown` must carry non-empty `options[]`, others must not. Invalid ŌåÆ 400.
5. **Lifecycle:** no hard delete (incidents pin types/versions). Retire = `isActive=false`
   (hidden from new-incident pickers; old incidents still resolve). System types can gain
   new versions, be renamed, and be deactivated, but their `key` and `isSystem` are
   immutable. Custom types have full control. Every write is audited.

---

## Goals / success criteria (the 1.4 Test Gate)

- **Editing a form creates v2 and leaves v1 intact** (proven against the DB ŌĆö v1 row
  byte-identical after the edit).
- **An old report still renders against its pinned version** (fetch v1 after v2 exists;
  v1's `schema` is unchanged and resolvable).
- Plus the recurring rules: tenant-scoped access only; fail-closed filter intact; leak
  suite still green with the two new tables; isolation allowlist unchanged
  (`{Plan, SharedIntelligenceEntry}`).

## A. Data model (Prisma)

Add tables 12 and 13 exactly per `schema.md`, plus the `ReportLane` enum. One migration.

```prisma
enum ReportLane { SECURITY SAFETY }

model ReportType {
  id        String     @id @default(cuid())
  tenantId  String
  key       String
  name      String
  lane      ReportLane @default(SECURITY)
  isSystem  Boolean    @default(false)
  isActive  Boolean    @default(true)
  createdAt DateTime   @default(now())

  tenant    Tenant              @relation(fields: [tenantId], references: [id], onDelete: Cascade)
  versions  ReportTypeVersion[]

  @@unique([tenantId, key])
  @@index([tenantId, lane])
}

model ReportTypeVersion {
  id           String   @id @default(cuid())
  tenantId     String
  reportTypeId String
  version      Int
  schema       Json     // Field[] ŌĆö see ┬¦B
  createdById  String
  createdAt    DateTime @default(now())

  reportType ReportType @relation(fields: [reportTypeId], references: [id], onDelete: Cascade)

  @@unique([reportTypeId, version])
  @@index([tenantId, reportTypeId])
}
```

`Tenant` gains `reportTypes ReportType[]`. Both are ordinary tenant-scoped models ŌĆö the
Stage 0 extension auto-stamps/filters `tenantId`; **not** added to the exempt allowlist.

> **Deviation note:** `schema.md` table 13 shows `incidents Incident[]` back-relations on
> both models, and `ReportType` shows `incidents Incident[]`. `Incident` is a 1.5 table, so
> those back-relations are added in 1.5 when `Incident` lands ŌĆö omitting them now keeps the
> schema compilable. Documented like the 1.1/1.2 deviations.

## B. Field-schema contract (`src/shared/report-schema.ts`)

A reusable Zod validator for a form's field list. 1.4 validates it on catalog writes; 1.5
will reuse it to interpret `fieldValues` at fill time.

```ts
import { z } from "zod";

export const fieldSchema = z
  .object({
    key: z.string().min(1),
    label: z.string().min(1),
    type: z.enum(["text", "dropdown", "photo", "yesno"]),
    required: z.boolean(),
    options: z.array(z.string().min(1)).min(1).optional(),
  })
  .superRefine((f, ctx) => {
    if (f.type === "dropdown" && !f.options)
      ctx.addIssue({ code: z.ZodIssueCode.custom, message: "dropdown requires options", path: ["options"] });
    if (f.type !== "dropdown" && f.options)
      ctx.addIssue({ code: z.ZodIssueCode.custom, message: "options only allowed on dropdown", path: ["options"] });
  });

export const formSchema = z
  .array(fieldSchema)
  .superRefine((fields, ctx) => {
    const keys = fields.map((f) => f.key);
    if (new Set(keys).size !== keys.length)
      ctx.addIssue({ code: z.ZodIssueCode.custom, message: "field keys must be unique within a form" });
  });

export type Field = z.infer<typeof fieldSchema>;
export type FormSchema = z.infer<typeof formSchema>;
```

Invalid field schema ŌåÆ the route returns 400 (via `validateBody` / explicit parse). Stored
to the DB as `Prisma.InputJsonValue`.

## C. Service (`src/modules/report-types/report-types.service.ts`)

All reads/writes go through `getScopedPrisma()` (tenantId auto-handled). Every mutating call
emits `audit.record`.

```ts
createType(input: { key: string; name: string; lane: ReportLane }, fields: FormSchema): Promise<{type, version}>
  // one $transaction: insert ReportType + ReportTypeVersion(version=1, createdById=ctx.userId)
  // duplicate [tenantId,key] ŌåÆ 409 (CONFLICT)
  // audit: "report_type.create"

addVersion(reportTypeId: string, fields: FormSchema): Promise<ReportTypeVersion>
  // load type (scoped; 404 if missing/other-tenant)
  // next = (max(version) for this type) + 1; insert new version row; old rows untouched
  // audit: "report_type.version.create"

updateTypeMeta(reportTypeId: string, patch: { name?: string; lane?: ReportLane; isActive?: boolean }): Promise<ReportType>
  // patch in place; key + isSystem are NOT accepted in the patch type (immutable)
  // works on system types too (rename/lane/retire allowed)
  // 404 if missing/other-tenant
  // audit: "report_type.update"

listTypes(opts?: { lane?: ReportLane; activeOnly?: boolean }): Promise<ReportType[]>
getType(reportTypeId: string): Promise<ReportType & { versions: ReportTypeVersion[] }>   // 404 if missing
getVersion(reportTypeId: string, version: number): Promise<ReportTypeVersion>            // 404 if missing
```

`createType`/`addVersion` validate `fields` through `formSchema.parse` (defense in depth ŌĆö
the route also validates). `isSystem` is never settable through the service (only the
seeder sets it true on insert).

## D. System seeder (`src/modules/report-types/system-types.ts`)

```ts
export const SYSTEM_REPORT_TYPES: ReadonlyArray<{ key: string; name: string; lane: ReportLane; fields: FormSchema }>
// six entries: theft (SECURITY), trespass (SECURITY), guest_dispute (SECURITY),
//              medical (SAFETY), hazard (SAFETY), lost_item (SECURITY)
// each ships a small, valid v1 field list (e.g. theft: location[text,req], item[text,req],
//   estimated_value[text], witness[text]; hazard: location[text,req], hazard_type[dropdown],
//   severity[dropdown,req]; medical: location[text,req], person_type[dropdown],
//   ambulance_called[yesno,req]; etc.)

export async function seedSystemReportTypes(tenantId: string): Promise<void>
// idempotent: for each entry, upsert ReportType by [tenantId,key] (isSystem=true);
//   if it has no version yet, insert version=1 (createdById = a system marker / null-safe).
// uses basePrisma with explicit tenantId (runs outside a request context, like token-cleanup
//   and the outbox relay) ŌåÆ its import is added to the base-client-boundary SANCTIONED list.
// Running it twice yields exactly six types, one v1 each ŌĆö no duplicates.
```

Called from `prisma/seed.ts` for each seeded tenant, and exported for future tenant
provisioning (Stage 2.1). `createdById` for seeded versions: since there is no acting user,
store a sentinel (`"system"`) ŌĆö `ReportTypeVersion.createdById` is a plain String (no FK to
User in the schema), so a sentinel is safe.

## E. Routes (`src/modules/report-types/report-types.routes.ts` + `.controller.ts`)

Mounted at `/api/report-types` in `app.ts`. Reads need auth; writes are admin-only
(`HOTEL_ADMIN`/`SUPER_ADMIN`), matching the properties module.

```
GET   /api/report-types               list      (authenticated)   ?lane=&activeOnly=
GET   /api/report-types/:id           detail + versions (authenticated)
POST  /api/report-types               create    (admin)   body {key,name,lane,fields}
POST  /api/report-types/:id/versions  new version (admin)  body {fields}
PATCH /api/report-types/:id           meta patch (admin)   body {name?,lane?,isActive?}
```

No `DELETE` (FK pin; retire via `PATCH {isActive:false}`). Controller exposes Zod body
schemas (`createTypeSchema`, `addVersionSchema`, `updateTypeSchema`) reusing `formSchema`,
validated by the existing `validateBody` middleware. Service errors (`409`, `404`) surface
through the existing `AppError` path.

## F. Testing (`src/tests/report-types/`)

1. **Version immutability (gate)** ŌĆö create a type (v1), `addVersion` with new fields ŌåÆ a
   v2 row exists AND the v1 row's `schema`/`createdAt` are byte-identical to before.
2. **Old version still resolves (gate)** ŌĆö after v2 exists, `getVersion(typeId, 1)` returns
   v1 with its original schema (an old incident pinned to v1 still renders).
3. **Field validation** ŌĆö `POST` / `addVersion` with: unknown `type`, duplicate `key`,
   `dropdown` without `options`, non-dropdown with `options`, empty `label` ŌåÆ each 400.
4. **Duplicate key** ŌĆö second `createType` with an existing `[tenantId,key]` ŌåÆ 409.
5. **System-type rules** ŌĆö a seeded system type: `addVersion` succeeds; `updateTypeMeta`
   rename/lane/`isActive` succeed; the patch type has no `key`/`isSystem` field so they
   cannot change (compile-time + a test asserting the stored `key`/`isSystem` are unchanged
   after a meta patch).
6. **Seeder idempotency** ŌĆö `seedSystemReportTypes(tid)` twice ŌåÆ exactly six types, one v1
   each, no duplicate-key error.
7. **Audit wiring** ŌĆö creating a type writes an `AuditLog` row `action="report_type.create"`
   with the right `actorUserId`/`tenantId` (reuses the 1.2 path).
8. **Leak suite** ŌĆö add `reportType` + `reportTypeVersion` to `READ_MODELS` in
   `isolation.test.ts` and seed one of each per tenant in `helpers/factories.ts` (tenant A
   never sees B's). Allowlist-lock still asserts exactly `{Plan, SharedIntelligenceEntry}`.
   Add the seeder file to the `base-client-boundary.test.ts` SANCTIONED importer list.

CI runs the suite serially against the migrated `hotelsec_test` DB with pristine output.
