# Stage 1.2 — Audit Log Design

**Date:** 2026-06-26
**Status:** Approved (brainstorm) — pending implementation plan
**Stage:** 1.2 (second slice of Stage 1, the Phase 1 operational core)
**References:** `new-md/checklist.md` §1.2 · `new-md/schema.md` table 23 (AuditLog) · `module-list.md` module 6 · builds on Stage 0 (`docs/STAGE-0-GUIDE.md`) and Stage 1.1 (`docs/STAGE-1-GUIDE.md`)

---

## Context

Stage 0 shipped an audit *seam*: `audit.record({action, entityType, entityId, metadata?})`
that stamps `tenantId / actorUserId / impersonatedBy / at` and writes to `console.info`
(`apps/api/src/modules/audit/audit.ts`). It is already called across the app — auth logout,
impersonation start/stop, and every property/building/floor/zone write in Stage 1.1.

Stage 1.2 turns that seam into a real, **append-only** `AuditLog` table: a tamper-evident
record of who did what, in which tenant, while impersonating whom. The `record()` signature
does not change, so no caller is touched — only the implementation swaps console for a DB
insert.

### Decisions locked during brainstorming
1. **Append-only enforcement:** DB triggers (BEFORE UPDATE / BEFORE DELETE `SIGNAL`) **plus**
   app discipline (audit module exposes only `record()`). Real tamper-evidence: even a raw
   query or a future bug cannot mutate/delete a row.
2. **impersonatedBy:** a dedicated `impersonatedBy String?` column (a documented deviation
   from locked table 23, as `archivedAt` was in 1.1) — queryable/indexable, since
   impersonation is security-critical.
3. **No read endpoint** in 1.2 — write path + immutability + tests only, matching the gate.
   A scoped timeline/search comes later (1.6 logbook is the user-facing timeline).
4. **IP capture:** thread `ip` into the request context so `AuditLog.ipAddress` is filled.
5. **Corrections, not deletions:** the log is append-only — nothing is ever edited or deleted.
   "This action was accidental/mistaken" is recorded by APPENDING a new audit row that links to
   the original via a nullable self-link `correctsId` (checklist A12). The original stays
   (that is the forensic value: the mistake AND its correction are both visible). 1.2 adds only
   the column + a test; no correction UI/endpoint yet.

---

## Goals / success criteria (the 1.2 Test Gate)

- A sensitive action writes an audit row.
- **No code path can edit or delete an audit row** (proven against the DB, not just the API).
- Plus the recurring rules: tenant-scoped access only; fail-closed filter intact; leak suite
  still green with the new table; isolation allowlist unchanged (`{Plan, SharedIntelligenceEntry}`).

---

## A. Data model (Prisma)

Add the `AuditLog` model (table 23 fields + the `impersonatedBy` deviation). One migration.

```prisma
model AuditLog {
  id             String   @id @default(cuid())
  tenantId       String
  actorUserId    String?            // who; null for system actions
  impersonatedBy String?            // platform admin id when acting via impersonation (deviation)
  action         String             // e.g. "property.create", "session.revoke"
  entityType     String             // loose pointer to any table…
  entityId       String             // …row id
  metadata       Json?
  ipAddress      String?
  at             DateTime @default(now())
  correctsId     String?            // self-link: this row corrects/flags an earlier audit row (A12)
  tenant      Tenant     @relation(fields: [tenantId], references: [id], onDelete: Cascade)
  corrects    AuditLog?  @relation("AuditCorrection", fields: [correctsId], references: [id])
  correctedBy AuditLog[] @relation("AuditCorrection")
  @@index([tenantId, entityType, entityId])
  @@index([tenantId, at])
}
```

`correctsId` points at another `AuditLog.id`. Append-only means the referenced row is never
deleted, so no `onDelete` rule is needed. 1.2 only adds the column/relation — no app code
writes `correctsId` yet (the "mark mistaken" flow is future work); a test confirms a correction
row links to its original.

`Tenant` gains the back-relation `auditLogs AuditLog[]`. Ordinary tenant-scoped model — the
Stage 0 extension auto-stamps/filters `tenantId`; it is NOT added to the exempt allowlist.

## B. Append-only triggers

`prisma migrate dev` creates the table; append raw SQL to the SAME migration file (Prisma
does not model triggers):

```sql
CREATE TRIGGER auditlog_no_update BEFORE UPDATE ON `AuditLog`
  FOR EACH ROW SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'AuditLog is append-only';
CREATE TRIGGER auditlog_no_delete BEFORE DELETE ON `AuditLog`
  FOR EACH ROW SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'AuditLog is append-only';
```

Any `UPDATE`/`DELETE` (scoped client, raw query, or bug) throws; `INSERT` still works. The
migration must apply identically on dev (`migrate dev`) and the test DB (`migrate deploy` via
global-setup), so the triggers exist everywhere.

## C. Test-cleanup interaction (non-obvious seam)

`resetDb()` (`apps/api/src/tests/helpers/test-db.ts`) deletes rows between tests via
`deleteMany` — a `DELETE`, which the trigger would block for AuditLog. Fix: in `resetDb`,
truncate AuditLog with `await basePrisma.$executeRawUnsafe('TRUNCATE \`AuditLog\`')` instead of
`deleteMany`. `TRUNCATE` is DDL and does **not** fire `BEFORE DELETE` triggers. AuditLog is a
child (FK → Tenant) referenced by nothing, so TRUNCATE is allowed. Run it among the first
cleanup statements (before the tenant delete). (Real retention/purge of old audit rows is a
Phase 3 concern, out of scope here.)

## D. `audit.record` rewrite (console → DB, same signature)

`apps/api/src/modules/audit/audit.ts`:

```ts
import { requireContext } from "../../shared/context/request-context.js";
import { getScopedPrisma } from "../../shared/prisma/index.js";

export const audit = {
  async record(input: { action: string; entityType: string; entityId: string; metadata?: unknown }): Promise<void> {
    const ctx = requireContext();
    await getScopedPrisma().auditLog.create({
      data: {
        action: input.action, entityType: input.entityType, entityId: input.entityId,
        metadata: input.metadata ?? undefined,
        actorUserId: ctx.userId ?? null,
        impersonatedBy: ctx.impersonatedBy ?? null,
        ipAddress: ctx.ip ?? null,
        // tenantId auto-stamped by the extension; `at` defaulted by the DB
      },
    });
  },
};
```

- `console.info` removed (Stage 0 tests silenced it; now real rows — output stays pristine).
- `requireContext()` (was `getContext()`): every current caller runs inside a request context,
  and an audit write with no context should fail-closed, not silently drop.
- Scoped client → `tenantId` stamped by the extension; the INSERT is allowed by the triggers.

## E. IP capture via context

- `apps/api/src/shared/context/request-context.ts`: add `ip?: string` to `RequestContext`.
- `apps/api/src/middleware/load-context.ts`: set `ip: req.ip` when opening the context.
- `audit.record` reads `ctx.ip`. If skipped, `ipAddress` simply stays null — but we include it
  (audit IP has forensic value).

## F. Testing

1. **Write path** — `POST /api/properties` (admin) → exactly one AuditLog row with
   `action="property.create"`, correct `actorUserId`, `tenantId`, non-null `at`.
2. **Immutability (the gate)** — `basePrisma.auditLog.update(...)`, `.delete(...)`, and
   `.deleteMany(...)` each reject (trigger `SIGNAL`). This proves true append-only, not just
   the API surface.
3. **Impersonation** — `POST /api/platform/impersonate` → row with `impersonatedBy` set and
   `actorUserId = "platform:<id>"`.
4. **Leak suite** — add `auditLog` to `factories.ts` (seed one row per tenant) and to
   `READ_MODELS` in `isolation.test.ts` (tenant A never sees B's rows). Allowlist-lock test
   still asserts exactly `{Plan, SharedIntelligenceEntry}`.
5. **Correction link** — insert an audit row with `correctsId` pointing at an earlier row;
   confirm the `corrects`/`correctedBy` relation resolves (and the original row is untouched).
6. **resetDb** — truncates AuditLog (per §C); full suite stays green with pristine output.

## G. Out of scope

- Read/timeline/search endpoint (later; 1.6 logbook).
- Retention / purge / anonymization of old audit rows (Phase 3).
- Auditing system/worker actions that run without a request context.
- Backfilling audit rows for past events.
